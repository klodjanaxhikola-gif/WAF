Access Governance

Roles are already defined and enforced

METRIX does not rely only on UI hiding. Access is enforced through role, page access, campaign access and backend guards.

super_admin

Full access, user governance, role assignment and manual overrides. Protected by OWNER_EMAIL fallback.

admin

Operational settings and full bypass for page/campaign access, without super_admin escalation rights.

tl

Role defaults like user; Working Hours only by assigned campaigns; workforce visibility limited to Agents.

fm

Working Hours only by assigned campaigns; visibility includes Agent and TL positions.

csdm

Working Hours only by assigned campaigns; visibility includes Agent, TL and FM positions.

trainer

Training Tracker editor by default; never has Working Hours access even if campaign access is assigned.

user

Dashboard and Leaves as viewer by default; Working Hours only through manual campaign assignment.

Manual overrides

Page and campaign overrides always win where applicable, managed through Users & Permissions.

RBAC implementedRole + Page Access + Campaign Access + Workforce Visibility

Permission Model

Enterprise access model by layer

1. Page Access

Controls normal pages with editor, viewer or no_access. Viewer hides write actions and backend writes return 403.

2. Campaign Access

Controls Working Hours and Training Hours. A user can be editor on one campaign and viewer or no_access on another.

3. Workforce Visibility

Filters visible employees by position for TL, FM and CSDM after campaign access is already granted.

Fail-closed for campaign access

No campaign permission means no Working Hours access for non-admin users. Campaign assignment is manual and deliberate.

Backend-enforced security

Direct API calls cannot bypass frontend state. Write routes check editor-level permissions before accepting changes.

Permission designSee, edit and export only what access allows

Validation Controls

Macro validation map — what is protected

Validation is not a single rule. It is a control framework covering data completeness, lifecycle consistency, date logic, eligibility, time conflicts, closures, leaves and permissions.

Control categories

  • Required fields and normalized values
  • Date ordering and valid lifecycle windows
  • Contract and campaign overlap prevention
  • Shared contract coverage by Dedicated contract
  • Working/Training Hours eligibility
  • Shift overlap and duplicate protection
  • Leave conflicts and closed-period blocking
  • Viewer/editor/no_access enforcement

Master data controls

Contracts

Required agent data, start/end logic, Dedicated overlap blocking and Shared coverage checks.

Campaigns

Required campaign metadata, lowercase project slug and overlapping campaign-period prevention.

Training

Training/Nesting/Production date validation and fixed phase/status/pass-status values.

Operational controls

Hours

Overnight shifts, split shifts, break handling, duplicate shift key and global overlap checks per agent.

Closure

Working Hours close by work date; Training Hours close by payment month.

Leaves

WH/Training Hours save is blocked when leave exists, and conflict reports catch later HR updates.

Validation layerFrontend feedback + backend enforcement

Working Hours Control

Working Hours is governed by contract, campaign, time and closure logic

1

Eligible agent

Agent is selected from active contracts and merged with Training Tracker status where relevant.

2

Authorized campaign

User must have campaign-level viewer/editor access for the project and campaign.

3

Valid shift

Split and overnight shifts are allowed, but overlaps and invalid times are blocked.

4

Open period

Workplan closure and month-close rules decide whether create/update/delete is allowed.

Global overlap scope

Overlap is checked by personal_id across projects and campaigns, not only inside the visible project.

Training Tracker integration

Pre-production trainees are hidden from Working Hours; post-production trainees need a valid contract to be editable.

Audit-safe behavior

Read-only states are shown for missing contracts or closed payment/work periods; viewing remains possible where appropriate.

Working Hours governanceEligibility + authorization + validation + closure

Future Direction

What comes next — without a timeline

The future direction is not about adding more screens. It is about transforming post-factum corrections into controlled, auditable workflows.

Contract versioning

History, restore and undo when contract updates are wrong or need formal rollback.

Leave / sick requests

Request submission, HR approval queue and approved records that automatically block WH entry.

WFM shift import

CSV import of planned shifts with validation, exception handling and controlled ownership.

Shift edit requests

TL-requested changes reviewed through validation and approval rather than uncontrolled direct edits.

Strategic principle

Permissions decide what a user can see. Validations decide what is technically acceptable. Requests decide what a user can change.

Major Platform Upgrade

Controlled change flows

This upgrade makes the next step very clear: sensitive operational changes no longer happen as free edits. They move through a structured request, review, validation, approval and audit cycle.

Upgrade logic Existing controls stay in place; a request lifecycle is added above them.
Flow A · Leave / Sick approval
1

Submit leave/sick request

User or TL inserts dates, leave type and reason.

2

HR review

HR approves or rejects the request with decision context.

3

Approved leave

The request becomes an official approved leave record.

4

WH blocked

Working Hours cannot be added on approved leave days.

Flow B · Shift change approval
1

CSV shift import

WFM uploads the planned shift baseline.

2

TL request edit

TL requests a change, not a full re-upload or uncontrolled edit.

3

Approval / validation

Rules check access, overlap, closure and leave conflict.

4

Audit trail

Original, requested and approved values are preserved.

Executive message: METRIX moves from preventing invalid input to governing the full lifecycle of operational change.
